# Requirements Document: Telegram Smart Assistant Bot

## Functional Requirements

### FR-1: Пользовательская аутентификация

**Описание**: Система должна аутентифицировать пользователей через Telegram Mini App с использованием initData

**Критерии приемки**:
- Система валидирует криптографическую подпись initData
- Система проверяет временную метку initData (не старше 1 часа)
- Система извлекает user_id и другие данные пользователя из initData
- При невалидном initData возвращается HTTP 401 Unauthorized
- Новые пользователи автоматически регистрируются в базе данных

**Приоритет**: Критический

### FR-2: Запись аудио в Mini App

**Описание**: Пользователь должен иметь возможность записывать аудио через браузерный интерфейс Mini App

**Критерии приемки**:
- Кнопка "Начать запись" активирует MediaRecorder API
- Во время записи отображается таймер и индикатор активности
- Кнопка "Остановить запись" завершает запись и сохраняет аудио
- Поддерживаются форматы: webm, ogg, mp3
- Максимальная длительность записи: 2 часа
- Максимальный размер файла: 100 MB
- При превышении лимитов отображается предупреждение

**Приоритет**: Критический

### FR-3: Создание конспекта из аудиозаписи

**Описание**: Система должна обрабатывать аудиозапись и создавать структурированный конспект

**Критерии приемки**:
- Аудиофайл загружается на сервер через API
- Задача обработки добавляется в очередь Redis
- Celery worker выполняет последовательно: STT → LLM → TTS
- STT транскрибирует аудио в текст через Whisper API
- LLM обрабатывает транскрипт согласно настройкам пользователя
- TTS синтезирует голосовое сообщение из конспекта
- Результат сохраняется в базу данных (notes_history)
- Пользователь получает уведомление в Telegram с голосовым сообщением и текстом
- Mini App отображает статус обработки в реальном времени

**Приоритет**: Критический


### FR-4: Настройки конспектирования

**Описание**: Пользователь должен иметь возможность настраивать параметры создания конспектов

**Критерии приемки**:
- Доступны стили изложения: detailed, brief, bullet_points, summary
- Доступны структуры: chronological, thematic, hierarchical
- Настройки включения: термины, даты, формулы (boolean)
- Настройка максимальной длины конспекта (100-10000 символов)
- Специальные режимы: cheatsheet, action_items, diarization, glossary
- Выбор языка обработки
- Настройки сохраняются в user_settings
- Настройки применяются при создании нового конспекта
- Snapshot настроек сохраняется вместе с конспектом

**Приоритет**: Высокий

### FR-5: История конспектов

**Описание**: Пользователь должен иметь доступ к истории созданных конспектов

**Критерии приемки**:
- Отображается список всех конспектов пользователя
- Для каждого конспекта показывается: заголовок, дата создания, длительность
- Возможность просмотра полного текста конспекта
- Возможность прослушивания голосового конспекта
- Возможность просмотра оригинальной транскрипции
- Пагинация списка (20 записей на страницу)
- Возможность удаления конспекта
- Поиск по заголовкам конспектов

**Приоритет**: Средний

### FR-6: Диалоговый помощник (текстовый режим)

**Описание**: Пользователь должен иметь возможность задавать вопросы текстом и получать ответы от LLM

**Критерии приемки**:
- Текстовое поле для ввода вопроса
- Отправка вопроса через API
- Система загружает последние N сообщений из истории (контекст)
- LLM генерирует ответ с учетом контекста и настроек
- Ответ отображается в интерфейсе чата
- Вопрос и ответ сохраняются в conversation_history
- Индикатор "печатает..." во время генерации ответа
- Поддержка markdown форматирования в ответах

**Приоритет**: Критический

### FR-7: Диалоговый помощник (голосовой режим)

**Описание**: Пользователь должен иметь возможность задавать вопросы голосом и получать озвученные ответы

**Критерии приемки**:
- Кнопка записи голосового вопроса
- Аудио транскрибируется через STT
- Транскрипция отображается в чате как вопрос пользователя
- LLM генерирует текстовый ответ
- Если озвучка включена в настройках - ответ синтезируется через TTS
- Голосовой ответ воспроизводится в интерфейсе
- Текстовая версия ответа также отображается

**Приоритет**: Высокий

### FR-8: Настройки диалога

**Описание**: Пользователь должен иметь возможность настраивать параметры диалогового помощника

**Критерии приемки**:
- Выбор длины ответа: short, medium, long
- Выбор стиля общения: formal, casual, technical
- Включение/выключение озвучки ответов
- Настройка количества сообщений в контексте (1-50)
- Выбор языка общения
- Настройки сохраняются в user_settings
- Изменения применяются к последующим сообщениям

**Приоритет**: Средний

### FR-9: История диалога

**Описание**: Система должна сохранять и отображать историю диалога пользователя

**Критерии приемки**:
- Все сообщения (user и assistant) сохраняются в conversation_history
- История отображается в хронологическом порядке
- Загрузка истории при открытии экрана диалога
- Возможность очистки истории диалога
- При очистке удаляются записи из базы данных
- Автоматическая подгрузка старых сообщений при скролле вверх

**Приоритет**: Средний


### FR-10: Система подписок

**Описание**: Система должна поддерживать различные тарифные планы с ограничениями

**Критерии приемки**:
- Три тарифа: Free, Basic, Premium
- Free: 10 конспектов/месяц, 100 диалоговых сообщений/месяц
- Basic: 50 конспектов/месяц, 500 диалоговых сообщений/месяц
- Premium: безлимитные конспекты и сообщения
- Проверка лимитов перед обработкой запроса
- При превышении лимита возврат HTTP 429 с информацией о тарифе
- Отображение текущего использования в профиле
- Сброс счетчиков в начале каждого месяца

**Приоритет**: Высокий

### FR-11: Обработка платежей

**Описание**: Пользователь должен иметь возможность оплатить подписку

**Критерии приемки**:
- Интеграция с Telegram Stars API
- Отображение доступных тарифов с ценами
- Кнопка "Оформить подписку" открывает платежную форму Telegram
- После успешной оплаты обновляется subscription_status
- Устанавливается subscription_end (текущая дата + период подписки)
- Пользователь получает уведомление об активации подписки
- Автоматическое продление подписки (если включено)
- Уведомление за 3 дня до окончания подписки

**Приоритет**: Высокий

### FR-12: Профиль пользователя

**Описание**: Пользователь должен иметь доступ к информации о своем профиле

**Критерии приемки**:
- Отображение имени пользователя и username из Telegram
- Отображение текущего тарифа и даты окончания подписки
- Статистика использования: количество конспектов, диалоговых сообщений
- Дата регистрации
- Кнопка перехода к настройкам
- Кнопка управления подпиской

**Приоритет**: Средний

### FR-13: Навигация в Mini App

**Описание**: Mini App должен иметь удобную навигацию между экранами

**Критерии приемки**:
- Нижняя панель навигации с 4 кнопками
- Экраны: Конспекты, Диалог, Профиль, Настройки
- Активный экран выделяется визуально
- Переключение между экранами без перезагрузки
- Сохранение состояния при переключении экранов
- Адаптивный дизайн для различных размеров экрана

**Приоритет**: Высокий

### FR-14: Индикаторы прогресса

**Описание**: Система должна информировать пользователя о статусе обработки задач

**Критерии приемки**:
- Индикатор загрузки при отправке аудио
- Статусы задачи: queued, processing, completed, failed
- Прогресс-бар с процентами выполнения (0-100%)
- Текстовое описание текущего этапа (STT, LLM, TTS)
- Polling статуса задачи каждые 2 секунды
- Уведомление при завершении обработки
- Отображение ошибки при неудачной обработке

**Приоритет**: Высокий

### FR-15: Обработка ошибок

**Описание**: Система должна корректно обрабатывать ошибки и информировать пользователя

**Критерии приемки**:
- Валидация размера и формата аудиофайлов
- Обработка ошибок внешних сервисов (STT, LLM, TTS)
- Retry логика с экспоненциальной задержкой (до 3 попыток)
- Понятные сообщения об ошибках для пользователя
- Логирование всех ошибок на сервере
- Уведомление администратора при критических ошибках
- Graceful degradation при недоступности сервисов

**Приоритет**: Критический


### FR-16: Экспорт конспектов в различные форматы

**Описание**: Пользователь должен иметь возможность экспортировать конспекты в различные форматы

**Критерии приемки**:
- Экспорт в текстовый файл (.txt)
- Экспорт в Markdown (.md)
- Экспорт в PDF с форматированием (заголовки, списки, выделение терминов) через fpdf2
- Экспорт в DOCX через python-docx
- Кнопка "Экспорт" в детальном просмотре конспекта
- Выбор формата экспорта из списка
- Скачивание файла через браузер (для PDF, DOCX, TXT, MD)
- Имя файла содержит заголовок и дату создания
- Асинхронная обработка экспорта для больших конспектов

**Приоритет**: Средний

### FR-17: Экспорт в Google Docs

**Описание**: Пользователь должен иметь возможность экспортировать конспекты в Google Docs

**Критерии приемки**:
- OAuth 2.0 авторизация при первом экспорте
- Открытие окна авторизации Google
- Сохранение refresh token в БД (зашифрованно)
- Создание нового документа в Google Docs
- Вставка конспекта с форматированием (заголовки, списки, выделение)
- Возврат ссылки на созданный документ
- Отправка ссылки в Telegram чат
- Автоматическое обновление access token при истечении
- Сохранение document_id в notes_history
- Возможность повторного экспорта (обновление существующего документа)

**Приоритет**: Средний

### FR-18: Экспорт в Google Sheets

**Описание**: Пользователь должен иметь возможность экспортировать структурированные данные в Google Sheets

**Критерии приемки**:
- Использование того же OAuth токена, что и для Google Docs
- Создание новой таблицы в Google Sheets
- Экспорт action items построчно (если есть в конспекте)
- Экспорт глоссария терминов (если есть в конспекте)
- Форматирование заголовков таблицы
- Возврат ссылки на созданную таблицу
- Отправка ссылки в Telegram чат
- Сохранение spreadsheet_id в notes_history

**Приоритет**: Низкий

### FR-19: Telegram Bot команды

**Описание**: Telegram Bot должен поддерживать базовые команды

**Критерии приемки**:
- /start - приветствие и кнопка открытия Mini App
- /help - справка по использованию бота
- /profile - информация о профиле и подписке
- /settings - быстрый доступ к настройкам
- Inline кнопки для навигации
- Отправка результатов конспектирования в чат

**Приоритет**: Средний

### FR-20: Настройки TTS

**Описание**: Пользователь должен иметь возможность настраивать параметры синтеза речи

**Критерии приемки**:
- Выбор голоса из доступных вариантов
- Настройка скорости речи (0.5x - 2.0x)
- Предпросмотр голоса (воспроизведение примера)
- Настройки применяются к конспектам и диалогу
- Сохранение в user_settings

**Приоритет**: Средний

## Non-Functional Requirements

### NFR-1: Производительность

**Описание**: Система должна обеспечивать приемлемое время отклика

**Критерии приемки**:
- API endpoints отвечают за < 500ms (без учета внешних сервисов)
- STT обработка: < 30 секунд для 10-минутной записи
- LLM обработка: < 20 секунд для транскрипта до 5000 токенов
- TTS синтез: < 15 секунд для текста до 2000 символов
- Загрузка Mini App: < 3 секунды
- Загрузка истории: < 1 секунды

**Приоритет**: Высокий

### NFR-2: Масштабируемость

**Описание**: Система должна поддерживать рост числа пользователей

**Критерии приемки**:
- Поддержка до 1000 одновременных пользователей
- Обработка до 100 задач конспектирования в час
- Обработка до 500 диалоговых сообщений в минуту
- Возможность горизонтального масштабирования API серверов
- Возможность добавления Celery workers
- Database read replicas для распределения нагрузки

**Приоритет**: Средний

### NFR-3: Безопасность

**Описание**: Система должна обеспечивать защиту данных пользователей

**Критерии приемки**:
- Все соединения через HTTPS
- Валидация initData с криптографической подписью
- API keys хранятся в переменных окружения
- Шифрование аудиофайлов в хранилище (AES-256)
- Rate limiting: 100 запросов в минуту на пользователя
- CORS политика ограничивает доступ к API
- Защита от SQL injection через ORM
- Защита от XSS в Mini App
- Санитизация пользовательского ввода

**Приоритет**: Критический

### NFR-4: Доступность

**Описание**: Система должна быть доступна для пользователей

**Критерии приемки**:
- Uptime: 99.5% (допустимый downtime: 3.6 часа в месяц)
- Graceful degradation при недоступности внешних сервисов
- Health check endpoints для мониторинга
- Автоматический restart при сбоях
- Backup базы данных каждые 24 часа
- Disaster recovery план

**Приоритет**: Высокий


### NFR-5: Удобство использования

**Описание**: Интерфейс должен быть интуитивно понятным и удобным

**Критерии приемки**:
- Адаптивный дизайн для мобильных устройств
- Поддержка темной и светлой темы (из настроек Telegram)
- Понятные иконки и подписи
- Минимальное количество кликов для основных действий
- Подсказки для новых пользователей
- Обратная связь на действия пользователя (анимации, уведомления)

**Приоритет**: Высокий

### NFR-6: Совместимость

**Описание**: Mini App должен работать на различных платформах

**Критерии приемки**:
- Поддержка iOS (Safari, Telegram iOS)
- Поддержка Android (Chrome, Telegram Android)
- Поддержка Desktop (Chrome, Firefox, Safari, Telegram Desktop)
- Корректная работа MediaRecorder API на всех платформах
- Fallback для неподдерживаемых функций

**Приоритет**: Высокий

### NFR-7: Локализация

**Описание**: Интерфейс должен поддерживать несколько языков

**Критерии приемки**:
- Русский язык (основной)
- Английский язык
- Автоматическое определение языка из настроек Telegram
- Возможность ручного переключения языка
- Локализация всех текстов интерфейса
- Локализация сообщений об ошибках

**Приоритет**: Средний (для v1.0)

### NFR-8: Мониторинг и логирование

**Описание**: Система должна обеспечивать мониторинг и логирование

**Критерии приемки**:
- Логирование всех API запросов
- Логирование ошибок с stack trace
- Метрики производительности (Prometheus)
- Дашборды для мониторинга (Grafana)
- Алерты при критических ошибках
- Трекинг использования внешних сервисов и затрат
- Error tracking через Sentry

**Приоритет**: Средний

### NFR-9: Конфиденциальность данных

**Описание**: Система должна соблюдать требования по защите персональных данных

**Критерии приемки**:
- GDPR compliance
- Возможность экспорта всех данных пользователя
- Возможность полного удаления данных пользователя
- Автоматическое удаление аудиофайлов через 30 дней
- Анонимизация данных для аналитики
- Политика конфиденциальности доступна в боте
- Согласие пользователя на обработку данных при первом запуске

**Приоритет**: Критический

### NFR-10: Безопасность OAuth и шифрование

**Описание**: Система должна обеспечивать безопасное хранение и использование OAuth токенов

**Критерии приемки**:
- OAuth 2.0 для Google интеграций с PKCE
- Шифрование refresh tokens в БД (AES-256)
- Автоматическое обновление access tokens при истечении
- Отзыв токенов при удалении пользователя
- Валидация redirect_uri для предотвращения атак
- Безопасное хранение encryption keys в переменных окружения
- Логирование всех OAuth операций

**Приоритет**: Критический

### NFR-11: Документация

**Описание**: Проект должен иметь полную документацию

**Критерии приемки**:
- README с инструкциями по установке и запуску
- API документация (OpenAPI/Swagger)
- Документация архитектуры
- Руководство по развертыванию
- Документация для разработчиков
- Пользовательская документация (FAQ)

**Приоритет**: Средний

## Technical Constraints

### TC-1: Технологический стек

**Backend**:
- Python 3.11+ с FastAPI
- PostgreSQL 15+ с Prisma Client Python
- Redis 7.2+ для очередей и кэша
- Redis Queue (RQ) для асинхронной обработки

**Frontend**:
- React 18+ или Vue 3+ или Svelte 4+
- TypeScript для типобезопасности
- Telegram Mini Apps SDK

**Infrastructure**:
- Vercel для FastAPI (serverless functions) и Mini App (static hosting)
- Pterodactyl для Python воркеров (тяжелые задачи)
- Docker для контейнеризации воркеров
- S3-compatible storage или Supabase Storage для файлов

**Ограничения Vercel**:
- Timeout: 60 секунд для serverless functions
- Размер пакета: максимум 250 MB
- Все длительные задачи должны делегироваться воркерам

### TC-2: Внешние сервисы

**Обязательные**:
- Groq API (Whisper-large-v3 для STT, llama-3.3-70b-versatile для LLM, TTS)
- Telegram Bot API
- Telegram Mini Apps API

**Для экспорта**:
- Google Docs API (для экспорта конспектов)
- Google Sheets API (для экспорта структурированных данных)
- Google OAuth 2.0 (для авторизации)

### TC-3: Ограничения платформы

- Максимальный размер файла в Telegram: 50 MB (для отправки через бота)
- Максимальная длительность голосового сообщения: нет ограничений, но рекомендуется до 10 минут
- Ограничения Telegram Bot API: 30 запросов в секунду
- Ограничения Mini App: работа только внутри Telegram

### TC-4: Лимиты внешних сервисов

**Groq API**:
- Очень низкая задержка (ключевое преимущество)
- Rate limits согласно тарифному плану
- Whisper-large-v3: поддержка различных форматов аудио
- llama-3.3-70b-versatile: контекст и токены согласно модели

**Google APIs**:
- OAuth 2.0 требует регистрации приложения
- Rate limits для Docs/Sheets API
- Квоты на создание документов

**Telegram Stars**:
- Минимальная сумма платежа
- Комиссия платформы

**Vercel**:
- Timeout: 60 секунд для serverless functions
- Размер пакета: 250 MB
- Bandwidth limits согласно тарифу

## Assumptions and Dependencies

### Assumptions

1. Пользователи имеют стабильное интернет-соединение для записи и отправки аудио
2. Пользователи используют современные версии Telegram (поддержка Mini Apps)
3. Большинство пользователей будут использовать мобильные устройства
4. Средняя длительность аудиозаписи для конспектирования: 10-30 минут
5. Средняя длина диалогового сообщения: 50-200 слов

### Dependencies

1. Доступность Groq API для STT/LLM/TTS
2. Стабильность Telegram Bot API и Mini Apps API
3. Наличие инфраструктуры для развертывания (Vercel, Pterodactyl, PostgreSQL, Redis)
4. Получение API ключей для всех внешних сервисов (Groq, Google)
5. Настройка платежной системы (Telegram Stars или ЮKassa)
6. Регистрация OAuth 2.0 приложения в Google Cloud Console
7. Node.js для генерации Prisma Client (в CI/CD)
